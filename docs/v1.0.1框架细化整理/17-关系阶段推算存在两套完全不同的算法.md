# 问题 17: 关系阶段推算存在两套完全不同的算法

## 问题类型
逻辑冲突 / 重复实现

## 涉及文件
- `backend/services/relationship.py` 第 57-72 行（`RelationshipService.derive_stage`）
- `backend/services/dynamic_analyzer.py` 第 238-284 行（`DynamicAnalyzer.update_relationship_stage`）

## 两套算法对比

### 算法 A（relationship.py — 四维均值法）
```python
def derive_stage(self, dimensions: dict) -> str:
    avg = (trust + familiarity + respect + comfort) / 4.0
    if avg >= 0.85: return "partner"
    if avg >= 0.65: return "mentor"
    if avg >= 0.45: return "friend"
    if avg >= 0.20: return "acquaintance"
    return "stranger"
```
- 基于 trust / familiarity / respect / comfort 四维度的均值
- 每次对话通过 `delta_map` 微调维度分数
- 阈值：0.20 / 0.45 / 0.65 / 0.85

### 算法 B（dynamic_analyzer.py — 消息数 + 情感效价法）
```python
async def update_relationship_stage(self, session_id, emotion, db) -> str:
    # 统计消息数，计算最近10条消息的平均 valence
    if message_count >= 30 and valence > 0.7: return "partner"
    elif message_count >= 20 and valence > 0.6: return "mentor"
    elif message_count >= 10 and valence > 0.5: return "friend"
    elif message_count >= 3: return "acquaintance"
    else: return "stranger"
```
- 基于消息总数 + 最近 10 条消息的平均情感效价（valence）
- 阈值完全不同：消息数 3/10/20/30，valence 0.5/0.6/0.7
- 还会自己开数据库连接（`SessionLocal()`），绕过 FastAPI 依赖注入

## 影响分析

1. **同一系统存在两种阶段推算结果**：不同调用方可能得到不同阶段
2. **算法 B 的阈值极低**：只需 3 条消息就能达到 `acquaintance`，30 条消息 + valence > 0.7 就能达到 `partner` — 与算法 A 的 0.85 均值要求差异巨大
3. **算法 B 自己管理数据库连接**：`SessionLocal()` + `db.commit()` + `db.close()` 绕过了 FastAPI 的依赖注入，存在连接泄漏风险
4. **不确定哪个是权威来源**：`learning_engine.py` 可能调用其中之一或两者都调用

## 调查结论（2026-04-14）

### 算法 B 是 100% 死代码

经全项目 `grep` 验证：

```
$ grep -rn "update_relationship_stage" backend/ --include="*.py"
backend/services/dynamic_analyzer.py:238:    async def update_relationship_stage(...)
```

- `dynamic_analyzer.update_relationship_stage` **仅在 dynamic_analyzer.py 中定义，全项目零调用**
- `learning_engine.py` 导入了 `DynamicAnalyzer` 但只使用 `relationship_service`（算法 A）
- `learning.py` 路由层也不调用算法 B

### 算法 A（`relationship.py` 四维均值法）是唯一实际运行的算法

调用链确认：
```
learning.py → LearningEngine.process_message() → relationship_service.update_dimensions() → derive_stage()
```

## ✅ 已确认修复方案

### 步骤 1：删除算法 B（死代码）

删除 `backend/services/dynamic_analyzer.py` 中的 `update_relationship_stage` 方法（约第 238-284 行，~46 行代码）。

**附带收益**：该方法内含的 `SessionLocal()` 自管数据库连接也一并消除（缓解 #18）。

### 步骤 2：确认 `derive_stage` 为唯一权威

`backend/services/relationship.py` 的 `RelationshipService.derive_stage` 是项目唯一的关系阶段推算实现，无需修改逻辑。

### 步骤 3：统一关系阶段常量（联动 #01, #03, #08）

确认唯一算法后，后续工作：
- 提取 `stage_map`（#01）和 `_default_relationship`（#03）为统一常量
- 后端常量放在 `backend/core/constants.py` 或 `relationship.py` 类属性
- 前端统一引用一处定义（#08）

### 影响面

| 文件 | 变更 | 风险 |
|------|------|------|
| `dynamic_analyzer.py` | 删除 ~46 行 | 无（零调用） |
| `relationship.py` | 无变更 | — |
| `learning_engine.py` | 可选清理 DynamicAnalyzer 导入 | 低 |

### 降级影响

- **#18（自管 DB 连接）**：删除算法 B 后，`dynamic_analyzer.py` 中只剩 `update_learner_profile` 方法需要处理自管连接，工作量减半
- **#01/#03/#08**：有了唯一权威算法，提取常量的方向明确
