## 原 #03: 版本号在 main.py 中硬编码重复

### 问题描述
`backend/main.py` 第 34 行和第 53 行分别硬编码 `version="1.0.0"`，发版时容易遗漏，且无法通过环境变量覆盖。

### 解决方案
1. `backend/core/config.py` 新增 `app_version: str = "1.0.0"` 字段
2. `main.py` 两处改为 `settings.app_version`
3. 支持 `.env` 文件覆盖：`APP_VERSION=1.1.0`

**PR**: #221 | **Issue**: #214

## 原 #04: character_sprites 是 sage_sprites 的冗余别名

### 问题描述
`start_learning` 响应中 `character_sprites` 值始终等于 `sage_sprites`，前端使用 `data.sage_sprites ?? data.character_sprites` 兼容两者，造成语义不清。

### 解决方案
通过提取 `_build_start_response()` 辅助函数统一返回字典。`character_sprites` 字段暂时保留为 `sage_sprites` 的 alias（前端 learning.ts 有 fallback 引用），标记为 legacy。

**PR**: #219 | **Issue**: #212

## 原 #05: start_learning 两条路径的返回字典几乎完全重复

### 问题描述
`start_learning` 函数内"复用已有 session"和"创建新 session"两条路径各自构造 12 个字段的返回字典，新增/修改字段时容易遗漏。

### 解决方案
提取 `_build_start_response()` 辅助函数，接收参数后统一构造返回字典。两条路径末尾统一调用，消除了约 20 行重复代码。

**PR**: #219 | **Issue**: #212

## 原 #08: 前端关系阶段标签在 types 和 constants 中重复定义

### 问题描述
`frontend/src/types/index.ts` 定义了 `STAGE_LABELS`，`frontend/src/constants/courseLevels.ts` 定义了 `RELATIONSHIP_STAGE_LABELS`，内容完全一致，仅命名不同。3 个组件引用 `STAGE_LABELS`，4 个组件引用 `RELATIONSHIP_STAGE_LABELS`。

### 解决方案
1. 删除 `types/index.ts` 中的 `STAGE_LABELS`（保留 `STAGE_COLORS` 等其他常量）
2. 将 3 个消费者（`RelationshipStageOverlay.vue`、`HudBar.vue`）迁移到 `RELATIONSHIP_STAGE_LABELS`
3. `RELATIONSHIP_STAGE_LABELS` 已在 #211 中从 `models.py` 同步到 `save.py` 和 `archive.py`，形成全栈统一

**PR**: #220 | **Issue**: #213

## 原 #13: user-profile 两个端点响应格式不一致

### 问题描述
`GET /user/profile` 直接返回 profile dict，`POST /user/profile/refresh` 包装在 `{ success, data, computed_at }` 中返回。前端 `client.ts` 被迫使用 `data.data ?? data` 兼容两种格式。

### 解决方案
1. `refresh` 端点改为：先清除缓存（`computed_at = None`），再调用 `get_user_profile` 重新计算
2. 直接返回 profile dict，与 `GET /user/profile` 格式完全一致
3. 前端 `data.data ?? data` 兼容代码可简化为直接使用 `data`（遗留清理见 #35）

**PR**: #220 | **Issue**: #213

## 原 #14: 活跃 session 查询逻辑在 learning.py 中重复三次

## 原 #15: 角色模板数据前后端重复定义且 key 不匹配

### 问题描述
前端 `SageCreateFlow.vue` 发送 `template_name='socrates'`，后端 `PERSONA_TEMPLATES` 使用中文 key `'苏格拉底型'`，导致 `PERSONA_TEMPLATES.get("socrates")` 永远 fallback 到 `"默认"` 模板。

当前因前端同时发送 `traits`（滑块值）走 `if traits is not None` 分支而未触发 bug，但使用预设模板（不发送 traits）时模板查找将失败。

### 解决方案
在 `backend/api/routes/archive.py` 的 `PERSONA_TEMPLATES` 中同时注册英文 key 和中文 key：
```python
PERSONA_TEMPLATES = {
    "socrates":      ["耐心", "追问型", "启发型"],
    "苏格拉底型":     ["耐心", "追问型", "启发型"],
    "einstein":      ["鼓励型", "探索型", "启发型"],
    "爱因斯坦型":     ["鼓励型", "探索型", "启发型"],
    ...
    "custom":        ["耐心", "启发型"],
    "默认":          ["耐心", "启发型"],
}
```

**PR**: #220 | **Issue**: #213

## 原 #19: SAVE_DIR 硬编码路径依赖 CWD

### 问题描述
`backend/services/save_file_manager.py` 硬编码 `SAVE_DIR = Path("data/saves")`，路径解析取决于进程启动目录（CWD），Docker/alembic 等不同启动方式可能导致存档写入不同位置。

### 解决方案
1. `config.py` 已有 `save_directory: str = "./saves"` 字段
2. `save_file_manager.py` 改为从 config 读取：`SAVE_DIR = Path(get_settings().save_directory)`
3. 支持 `.env` 覆盖：`SAVE_DIRECTORY=/data/saves`

**PR**: #221 | **Issue**: #214
