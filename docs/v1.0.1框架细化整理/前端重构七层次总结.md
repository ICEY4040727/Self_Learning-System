## 2. 层次一：API 调用层统一

**关联问题**：#29（API 模块未被使用）、#32（worldStore 未被使用）

### 2.1 现状量化

```
api/client.ts      → userProfileApi (0 引用), reportApi (0 引用)
api/character.ts   → characterApi   (0 引用)
api/memory.ts      → memoryApi      (1 引用 — Learning.vue)
stores/world.ts    → useWorldStore  (0 引用)

Views 直接 import client → 12+ 文件
Views 使用 useWorldStore → 0 文件
Views 使用 characterApi  → 0 文件
Views 使用 reportApi     → 0 文件
```

**结论**：存在两层封装（API 模块 + Store），但 Views 全部绕过，直接裸调 `client`。

### 2.2 决策：二选一

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **A: 用 Store 模式** | 跨组件状态共享、缓存、统一 loading/error | 需重构所有 View 的数据获取逻辑 | ⭐⭐⭐ |
| **B: 用 API 模块** | 轻量、改动小、仅封装路径和参数 | 无法跨组件共享状态 | ⭐⭐ |

### 2.3 方案 A 实施步骤（推荐）

```
Step 1: 扩展 stores/world.ts
  - 补充 errorMessage/showError 逻辑
  - 补充 Courses/Characters 嵌套状态管理

Step 2: 迁移 Views（按依赖顺序）
  - Worlds.vue → useWorldStore().fetchWorlds()
  - WorldDetail.vue → useWorldStore().fetchCourses() / fetchWorldCharacters()
  - Character.vue → useWorldStore().fetchCharacters()

Step 3: 删除 api/character.ts 中的重复逻辑
  - characterApi 的方法已在 Store 中重新实现
  - 保留 client.ts 作为基础配置（拦截器、baseURL）

Step 4: 回归测试
  - Worlds 列表页加载
  - WorldDetail 课程/角色加载
  - Character 列表页加载
```

### 2.4 方案 B 实施步骤（备选）

```
Step 1: 让 Views 使用已有的 API 模块
  - Character.vue → import { characterApi } from '@/api/character'
  - 补充 worldApi 模块（当前缺失）

Step 2: 删除 stores/world.ts（139 行死代码）

Step 3: 回归测试
```

### 2.5 验收标准

- [ ] 所有 API 调用统一经过封装层（Store 或 API 模块）
- [ ] 无 View 直接 `import client from '@/api/client'`（仅封装层内部使用）
- [ ] `characterApi`、`userProfileApi`、`reportApi` 要么被使用要么被删除
- [ ] `useWorldStore` 要么被使用要么被删除

## 4. 层次三：通用逻辑抽取为 Composables

**关联问题**：#33（error-toast 重复实现）

### 4.1 现状量化

**重复模式**（6 个文件）：

```typescript
// 每个文件都重复以下 3 件套
const loading = ref(false)
const errorMessage = ref('')

const showError = (error: unknown) => {
  errorMessage.value = parseApiError(error)
  setTimeout(() => (errorMessage.value = ''), 4000)  // ⚠️ 有的没有这行
}
```

**散布位置**：
- `views/WorldDetail.vue` — `error-toast` class + `showError()` + 4s 清除
- `views/Worlds.vue` — `error-toast` class + `showError()` + 4s 清除
- `views/Home.vue` — `error-toast` class + 无自动清除
- `components/KnowledgeGraph.vue` — `error` class + 无自动清除
- `components/TimelineTree.vue` — `error` class + 无自动清除
- `components/galgame/CheckpointPanel.vue` — `error` class + 无自动清除

**不一致之处**：
- class 名：`error-toast` vs `error`
- 清除行为：4s 自动清除 vs 手动清除 vs 永不清除
- CSS 实现：每个文件 scoped style 中独立定义

### 4.2 实施步骤

```
Step 1: 创建 composables/useAsync.ts
  export function useAsync() {
    const loading = ref(false)
    const errorMessage = ref('')
    
    const showError = (error: unknown) => {
      errorMessage.value = parseApiError(error)
      setTimeout(() => (errorMessage.value = ''), 4000)
    }
    
    const clearError = () => { errorMessage.value = '' }
    
    async function run<T>(fn: () => Promise<T>): Promise<T | undefined> {
      loading.value = true
      errorMessage.value = ''
      try { return await fn() }
      catch (e) { showError(e); return undefined }
      finally { loading.value = false }
    }
    
    return { loading, errorMessage, showError, clearError, run }
  }

Step 2: 创建全局 ErrorToast 组件
  components/ErrorToast.vue
  - 统一使用 error-toast 样式
  - props: message, duration
  - 自动淡入淡出动画

Step 3: 迁移各文件
  - 替换各自的 loading/errorMessage/showError 为 useAsync()
  - 替换各自的 <p class="error-toast"> 为 <ErrorToast>
  - 删除各文件 scoped style 中的 .error-toast / .error 定义

Step 4: 回归测试
  - 各页面的错误提示正常显示和消失
  - loading 状态正确切换
```

### 4.3 验收标准

- [ ] `useAsync` composable 被所有需要 loading/error 的组件使用
- [ ] 全局 `ErrorToast` 组件替代所有 inline error 显示
- [ ] 错误行为统一（4s 自动清除）
- [ ] 无组件内局部定义 `.error-toast` 或 `.error` CSS

## 10. 风险评估与回滚策略

### Phase 1 风险
