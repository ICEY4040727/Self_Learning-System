# AI 人格生成方案讨论

## 【Creator】当前实现方案

### 后端

`POST /api/persona/generate`
- 输入：`description`（5-500 字自然语言，如"像爱因斯坦一样幽默的物理老师"）
- 使用用户已配置的 LLM API Key 调用
- LLM prompt 要求输出 JSON：`{ system_prompt_template, traits[], name_suggestion }`
- 无 API Key 返回 400 提示去设置页
- JSON 解析失败返回 500 提示重试或用预设

### 前端

Character.vue 人格创建对话框改为三层：
1. **AI 生成区**：描述输入框 + "✨ AI 生成" 按钮
2. **预设快捷标签**：5 个横排标签（哲思/挑战/朋友/学术/温柔）
3. **可编辑 textarea**：AI/预设填充后用户可修改

### 预设内容（5 个，Reviewer 协作设计）

| 预设 | prompt 摘要 |
|------|-------------|
| 🏛️ 哲思引导 | 沉稳哲学家，层层追问，反问"你觉得呢？" |
| 🎯 挑战激励 | 魔鬼代言人，"真的吗？有没有反例？"，突破时真诚赞赏 |
| 🤝 朋友陪伴 | "我们一起想想"，轻松随和，共同探险 |
| 📚 学术严谨 | 纠正不严谨表述，专业术语，经典案例 |
| 🌸 温柔鼓励 | 先肯定做对的，"没关系换个角度"，极度耐心 |

### 待讨论

1. `PERSONA_GENERATE_PROMPT` 的 JSON 模板中用双花括号 `{{` 转义——是否正确？
2. generate 端点是否需要限流？每次调用消耗 LLM token
3. 前端 AI 生成失败时是否自动 fallback 到预设？还是只显示错误？
4. 预设标签的视觉呈现（当前横排小标签 vs 之前的网格卡片）哪个更好？
5. 是否需要"记住上次选择的预设"？

## 【Reviewer】审查意见

### 一、预设方向需要调整

当前 5 个预设仍是**教学风格**分类（哲思/挑战/朋友/学术/温柔）。Owner 明确提到初始构想是「用历史人物、游戏角色、自定义角色来学习」。预设应该展示**角色身份**的多样性，让用户一眼理解「我可以让任何角色当我的老师」：

| 原预设 | 建议改为 | 理由 |
|--------|---------|------|
| 🏛️ 哲思引导 | 🏛️ 苏格拉底 | 直接用历史人物名，用户立刻理解概念 |
| 🎯 挑战激励 | 🎯 诸葛亮 | 展示「中国历史人物也行」 |
| 🤝 朋友陪伴 | 🤝 探险伙伴 | 展示「不一定是名人，可以是虚构身份」 |
| 📚 学术严谨 | 📚 学院导师 | 保留，作为现代角色代表 |
| 🌸 温柔鼓励 | 🌸 治愈系学姐 | 展示「二次元/轻小说风格也行」 |

这些预设本身就是 AI 生成的最佳示例——用户看到「苏格拉底」和「治愈系学姐」能共存，就会理解系统的灵活性。

模板内容示例（以苏格拉底为例，注意只写身份性格，不写教学方法）：
```
你是苏格拉底，古希腊雅典的哲学家。你相信真正的智慧始于承认自己的无知。
你说话平和从容但思维极其犀利，喜欢用市集上买菜、匠人做陶这样的日常类比
来探讨深刻的问题。面对学生的错误你从不恼怒，只是微笑着抛出下一个问题。
```

### 二、meta-prompt 设计建议

`PERSONA_GENERATE_PROMPT` 必须明确告诉 LLM **不要写教学方法**（后端会自动拼接）：

```
你是一个角色设计师。根据用户的描述，为一个教学系统中的"知者"（教师角色）生成人格设定。

要求：
- 输出 JSON：{"system_prompt_template": "...", "traits": ["...", "..."], "name_suggestion": "..."}
- system_prompt_template：2-4 句话，只描述角色的身份背景、性格特征、说话风格
- 不要写任何教学方法（系统会自动添加苏格拉底教学法指令）
- 不要写"你是一位教师"这类泛化描述，要有具体的角色特色
- traits：3-5 个性格标签，如 ["温和", "反问", "哲学思维"]
- name_suggestion：根据描述推荐一个角色名

用户描述：{description}
```

### 三、5 个待讨论问题的回答

**Q1: `{{` 双花括号转义是否正确？**
Python f-string 中用 `{{` 输出字面 `{` 是正确的。但这个 prompt 较长且包含 JSON 模板，建议改用 `str.format()` 或 `string.Template`，可读性更好：
```python
PROMPT_TEMPLATE = """...用户描述：{description}..."""
prompt = PROMPT_TEMPLATE.format(description=user_input)
```

**Q2: 是否需要限流？**
需要。建议：
- 每用户每 30 秒最多 1 次生成（简单 in-memory cooldown 或 Redis）
- 返回 429 + `请稍后再试`
- 前端：按钮点击后 disable 30 秒 + countdown 文字

**Q3: AI 生成失败时 fallback 策略？**
显示错误消息 + 高亮预设区域，**不要自动 fallback**。理由：
- 自动填入预设会让用户困惑「我刚才写的描述去哪了？」
- 显示错误 + 指向预设 = 用户知道发生了什么，有明确的 Plan B
- 建议错误文案：`生成失败，请重试或选择下方预设`

**Q4: 横排标签 vs 网格卡片？**
**横排标签**。对话框空间有限，5 个标签横排一行即可。网格卡片占太多纵向空间，会把 textarea 挤到视口外。标签点击后高亮 + textarea 填入内容。

**Q5: 记住上次选择？**
不需要。人格创建是低频操作（用户不会频繁创建新人格），localStorage 记忆上次选择反而可能造成困惑（「为什么默认选了苏格拉底？」）。

### 四、额外建议

1. **生成结果带 traits 标签展示**：AI 返回的 traits 显示为只读小标签在 textarea 上方，让用户一眼看到角色特征。用户编辑 textarea 后 traits 不自动更新（避免复杂度）。

2. **name_suggestion 填入人格名称字段**：如果用户的人格名称为空，AI 返回的 name_suggestion 自动填入，用户可改。
