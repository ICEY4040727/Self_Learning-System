# 本地覆盖远程后的重启动规划（Reviewer 视角）

> **前提**：今天将本地代码直接推送至远程 main，如有冲突以本地覆盖远程为准。
> 覆盖之前的所有历史 issue / PR 不再整理，全部作废，从覆盖后的状态作为 v1.0.0 新基线。
>
> 作者：Reviewer
> 日期：2026-04-09

---

## 一、覆盖当天（D0）—— 建立新基线

### 1. 冻结一份"新 main 真相快照"
- 覆盖后立刻执行：
  - `git log -1`
  - `alembic current`
  - `backend/` 目录树
  - `frontend/src/views` 列表
- Reviewer 据此重写 `memory/project_actual_state.md`，旧的 5 天前快照作废。
- 重点确认：World / Course / WorldCharacter 模型是否随本地代码进入 main，这决定后面所有 issue 是否还成立。

### 2. 关掉所有过时 issue / PR
- #124–#133、#163–#172、#159–#160 全部 close。
- 关闭说明统一写："pre-v1.0.0 baseline reset, 重新分诊"。
- 不是清理，是**作废**，避免新工作和旧 issue 互相引用造成混乱。

### 3. 打 tag
- 名称建议：`v1.0.0-baseline` 或 `pre-v1.0.0-merge`。
- 用途：万一覆盖后发现丢东西可回溯。

---

## 二、新基线后第一周（D1–D7）—— 重新分诊

### 4. Reviewer 全量走查新 main，输出 Gap 清单
对照三个来源逐项比对：
- `CLAUDE.md` Roadmap Phase 5
- `idea/idea.md` 设计愿景
- `docs/v1.0.0前后端联调修复/世界_课程_角色_表单设计.md` 联调文档

每条 gap 落成新 issue，统一打 `v1.0.0` milestone，重新标 P0 / P1 / P2。

### 5. 核心三问必须有答案（Reviewer 验证，不听 Creator 口述）
- `memory.py` 是否还是 stub？知识检索接没接？
- World / Course / Character 三表单的前后端字段是否一致？
- Alembic 迁移链是否干净，能否从空库一键 `upgrade head`？

### 6. 建立"新基线后"的 PR 规矩
- 一个 PR 一件事，禁止再出现 `chore: 恢复代码 + 清理` 这种混合 commit。
- PR 模板强制写：
  - 影响的表
  - 影响的路由
  - 是否需要迁移
  - 测试如何跑
- 由 Reviewer 在每次 review 时把关。

---

## 三、第二周起（D8+）—— 推进 v1.0.0 收尾

### 7. 按依赖顺序排队
Reviewer 给出顺序，Creator 按序产出 PR：

1. **DB 层**：迁移补齐 + FK `nullable=False` + smoke test
2. **Service 层**：`memory.py` JSON 知识检索实装
3. **API 层**：WorldCharacter / Course materials CRUD 补完
4. **前端**：联调文档里列的表单逐个对齐
5. **端到端**：存档读档全链路、苏格拉底引导真实跑通一节课

### 8. 测试基线
- 给 `learning_engine`、`dynamic_analyzer`、`save` 路由各补一个最小回归测试。
- CI 至少跑 `pytest` + `npm run build`，避免主干随时坏。

### 9. Docker / 部署对齐
- `CLAUDE.md` 描述 PostgreSQL + ChromaDB + Neo4j，实际是 SQLite。
- 要么改文档要么补 compose，**不能再放任不一致**，否则下一次"实际状态 vs 文档"又会偏离。
